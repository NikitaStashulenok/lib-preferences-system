# Feature Gap Analysis & Implementation Plan (2026-03-11)

Этот документ фиксирует текущее состояние проекта и план доработок по замечаниям:

1. Пагинация в каталогах.
2. Невалидные пароли seed-пользователей.
3. `PATCH /api/v1/users/me` возвращает `400`.
4. Нет полноценной обработки ошибок на странице профиля.
5. UX профиля/аватаров: режим редактирования, 1 аватар, переименование `fullName -> nickname`, валидация.
6. Переработка рекомендаций (системные + пользовательские), форма оценки/отзыва при возврате, пагинация рекомендаций, новый UI выбора.
7. Рассылка доступов библиотекарям по email (SMTP).
8. Расширенный кабинет библиотекаря и админа: пользователи/CRUD/фильтры/пагинация/confirmation modal.
9. Страница деталей книги: средняя оценка, комментарии, права на редактирование/удаление.

---

## 0) Что уже есть сейчас (факт)

### Backend
- Каталог на backend уже возвращает `Page<Book>` через endpoint `/api/v1/books` с параметрами `page` и `size`.
- Рекомендации уже paged (`/users/{id}/recommendations` принимает `page/size`), но текущая логика использует только первый жанр и первого автора из профиля и фильтрует по `AND`.
- Есть сущности `Rating` и `Review`, и отдельные endpoints на создание рейтинга/отзыва.
- Возврат книги реализован отдельным endpoint `/loans/{id}/return`, но без обязательного шага "оценка + отзыв".
- Админские endpoint-ы ограничены: просмотр списка пользователей и смена ролей; полного CRUD нет.
- Email-функционал/SMTP сейчас отсутствует.

### Frontend
- Каталог и рекомендации показывают данные из `Page`, но не имеют UI-контролов пагинации.
- Профиль отправляет `PATCH /users/me` целиком, но не показывает серверные сообщения об ошибке.
- На странице профиля жёстко отображаются 3 аватара одновременно; выбора/сохранения одного аватара нет.
- В рекомендациях формы жанров/авторов требуют минимум 1 жанр и 1 автора одновременно.
- Страницы деталей книги (`/books/:id`) нет.

## 1) Замечание: «Нет пагинации в каталогах»

### Root cause
Backend уже paged, но frontend фиксирует `page=0` и не даёт переключать страницы.

### План изменений
- FE: добавить универсальный `Pagination` компонент с `page`, `size`, `totalPages`, `onPageChange`.
- FE: на `CatalogPage` управлять `page` в state и передавать в `useBooksQuery`.
- FE: сбрасывать `page` в `0` при изменении фильтров.
- FE: добавить selector размера страницы (10/20/50).

### Acceptance
- Пользователь может листать каталог по страницам.
- После изменения фильтра открывается 1-я страница.

## 2) Замечание: «У seed-пользователей невалидные пароли»

### Root cause
Сейчас в SQL-миграциях пароли задаются жёсткими bcrypt-хэшами. Если хэш повреждён/несовместим с encoder-стратегией, вход ломается.

### Рекомендуемый способ (безопасный)
Перенести создание demo-пользователей из SQL в Java bootstrap-компонент (например, `CommandLineRunner` для dev-профиля):
- пароль хранить в конфиге (`APP_DEMO_PASSWORD`),
- хэш рассчитывать через `PasswordEncoder` во время старта,
- upsert-логика по email.

### План
1. В SQL оставить только доменные сущности (книги, рейтинги, займы), а users/roles сидить в Java для `dev`.
2. Добавить endpoint/лог старта, который печатает demo-логины (без хэша).
3. Документировать demo-учётки и пароль в `backend/README.md`.

### Acceptance
- Любой seed-пользователь может войти одинаковым паролем из env.
- Нет hardcoded bcrypt в миграциях для users.

## 3) Замечание: `PATCH /users/me` → 400

### Вероятная причина
Фронтенд отправляет неочищенные значения, а backend валидирует поля (`@Email`, `@Pattern` для телефона). При невалидном `phoneNumber`/`email` backend возвращает `400`.

### План
- FE:
  - нормализовать payload перед отправкой (`'' -> null` для опциональных полей),
  - показывать `error.response.data` (валидационные детали),
  - добавить field-level messages.
- BE:
  - добавить единый `@RestControllerAdvice` для формата ошибок (`code`, `message`, `fieldErrors[]`).

### Acceptance
- Ошибки валидации отображаются рядом с полями.
- Пользователь видит причину `400`, а не общий текст.

## 4) Замечание: «Нет обработки ошибок на странице профиля»

### Acceptance

## 5) Замечание: профиль (редактирование, 1 аватар, `nickname`, валидация)

### План UI/UX
- По умолчанию профиль в read-only.
- Кнопка-карандаш «Редактировать» включает edit-mode.
- В edit-mode поля активны + кнопки «Сохранить» / «Отменить».
- Аватар:
  - показывать 1 текущий avatar,
  - кнопка изменения открывает модал/дропдаун выбора иконки (стиль как на референсе Tailwind select),
  - выбранный avatar сохраняется в профиле.

### План данных
- Переименовать поле отображения `fullName` -> `nickname` (миграция + DTO + фронт).
- Валидация:
  - `nickname`: 2..50, буквы/цифры/пробел/`_-.`,
  - `phone`: E.164 light,
  - `birthDate`: не в будущем.

### Acceptance
- В профиле отображается один аватар.
- Редактирование доступно только после клика по карандашу.
- Поле называется `nickname` во всех слоях (DB/API/UI).

## 6) Замечание: рекомендации + оценка/отзыв при возврате + теги системные/пользовательские

### Целевая модель
Рекомендации = объединение двух источников:
1. **Системные**: по поведению пользователя (взял -> вернул -> высоко оценил).
2. **Пользовательские**: по ручным предпочтениям (жанры/авторы), оба поля опциональны.

### Изменения backend
- Добавить таблицу/слой агрегатов интересов (genre/author weights).
- При возврате книги запускать workflow:
  - статус RETURNED,
  - открыть возможность оставить рейтинг (обязательно) и отзыв (опционально),
  - если рейтинг >= порога (например, 4), увеличивать вес жанров/автора книги.
- Новый endpoint рекомендаций с breakdown:
  - `GET /users/{id}/recommendations?source=all|system|user&page=&size=`
  - ответ: `items[]` с полем `sourceTags: ["SYSTEM", "USER"]`.

### Изменения frontend
- После `Return` показывать форму: stars + optional review.
- На странице рекомендаций:
  - фильтр источника (`Все / Системные / Пользовательские`),
  - теги на карточках,
  - пагинация,
  - UI выбора предпочтений в стиле улучшенного single/multi-select (как в предоставленном референсе).
- Убрать обязательность одновременного выбора и жанра, и автора.

### Acceptance
- Возврат книги включает шаг оценки.
- Рекомендации разделяются по источникам и пагинируются.

## Технические риски и контроль

- **Совместимость миграций**: rename `full_name -> nickname` потребует аккуратной миграции и обратной совместимости DTO.
- **Нагрузка на рекомендации**: для системных рекомендаций нужен индексируемый расчет веса (не делать тяжелые запросы на лету).
- **Безопасность email-инвайтов**: не хранить SMTP креды в репозитории, только env/secrets.
- **UX регрессии**: обязательны e2e сценарии: login, profile edit, return+rate, recommendations.
